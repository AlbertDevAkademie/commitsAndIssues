Commit-Typen und ihre Bedeutung

Hier eine Übersicht der verschiedenen Commit-Typen und deren Verwendung:

Info: Link 1 / Link 2
Commit-Typ Bedeutung Wann verwenden?
feat Funktionale Neuerungen Verwende diesen Commit-Typ, wenn neue Features hinzugefügt werden, wie z.B. das Hinzufügen von Funktionen für die Navigation von Bildern oder den Vollbildmodus.
fix Fehlerbehebungen Nutze diesen Typ für Bugfixes, wie z.B. das Beheben von Problemen mit der Skalierung von Bildern auf mobilen Geräten oder das Lösen von Flickern beim Laden von Bildern.
style Visuelle Anpassungen Verwende style, wenn du nur das Design änderst, wie z.B. das Verbessern der Hover-Effekte oder das Aktualisieren der Farbgebung für besseren Kontrast.
docs Dokumentation Verwende diesen Commit-Typ, wenn du Änderungen an der Dokumentation vornimmst, z.B. das Hinzufügen von Features oder die Aktualisierung des README mit Installationsschritten.
build Änderungen an Build-Prozessen Dieser Commit-Typ wird für Änderungen am Build-Prozess oder an Abhängigkeiten verwendet, z.B. beim Aktualisieren der Abhängigkeiten für bessere Performance.
refactor Code-Refactoring Verwende refactor für Änderungen am Code, die keine funktionalen Änderungen bewirken, wie z.B. das Optimieren der Logik für das Vorladen von Bildern oder das Bereinigen von CSS-Klassen.
test Fügt Tests hinzu oder aktualisiert bestehende Tests Wird verwendet, wenn neue Tests für Funktionen hinzugefügt oder bestehende Tests angepasst werden.

GitHub Keywords für Issues

Hier eine kleine Übersicht, wann man welches Keyword verwenden sollte:

Info: https://github.com/orgs/community/discussions/147722
Keyword Bedeutung Wann verwenden?
Closes #1 Schließt das Issue endgültig Allgemein empfohlen für alle abgeschlossenen Issues
Fixes #1 Zeigt, dass ein Bug behoben wurde Nur für Bugfixes geeignet
Resolves #1 Zeigt, dass das Problem gelöst ist Kann für alle Issues verwendet werden, nicht nur Bugs

Hinweis: Die Verwendung dieser Keywords hilft, Issues automatisch zu schließen und die Historie sauber zu halten.

Was ist Semantic Versioning (SemVer)?

Semantic Versioning 2.0.0

Semantic Versioning (SemVer) verwendet ein drei-stufiges Versionssystem:

MAJOR.MINOR.PATCH → 1.0.0

Versionstypen

  • MAJOR (1.x.x) → Große Änderungen, die inkompatibel sind (Breaking Changes)
  • MINOR (x.1.x) → Neue Features, aber abwärtskompatibel
  • PATCH (x.x.1) → Kleine Bugfixes, keine neuen Features

Würde man nur mit 1, 2, 3 arbeiten, wäre es schwer zu erkennen, ob es sich um einen kleinen Bugfix oder ein großes Update handelt.

Wie geht man vor?

Schritt 1: Die erste stabile Version veröffentlichen

Version beginnt bei 1.0.0 → Die erste richtige, stabile Version.

Vorher gibt es eventuell Alpha/Beta-Versionen → 0.x.x oder 1.0.0-beta

Schritt 2: Kleinere Features und Bugfixes einfügen

  • 1.0.1 → Kleiner Bugfix (z. B. ein Fix für falsche Bildgrößen)
  • 1.1.0 → Neues Feature, aber keine Breaking Changes (z. B. Slideshow hinzugefügt)

Schritt 3: Große Änderungen machen

Wenn sich die API oder Funktionen stark ändern und alte Dinge nicht mehr funktionieren → 2.0.0

Beispiel: Fotogalerie Versionierung

1.0.0 → Erste stabile Version

Die erste stabile Version der Fotogalerie, mit grundlegenden Funktionen.

1.0.1 → Kleiner Bugfix

Ein Bugfix, z. B. ein Fix für falsche Bildgrößen.

1.1.0 → Neues Feature

Ein neues Feature wird hinzugefügt, z. B. eine Slideshow.

2.0.0 → Große Änderung

Eine große Änderung, z. B. eine komplette Umstellung auf ein neues Framework.
